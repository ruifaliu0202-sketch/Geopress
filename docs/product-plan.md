# 产品功能节点与实施计划

## 产品定位

Geopress 是一个面向个人和公司的多租户内容自动发布平台。核心闭环是：

```text
登录用户 -> 进入个人/公司工作区 -> 维护知识库和媒体账号 -> 输入关键词生成内容草稿 -> 编辑审核 -> 创建发布计划 -> 执行发布任务 -> 回收发布结果
```

AI 能力可以后置接入，但产品框架必须先把上下文、知识库、媒体账号、内容生命周期和计划任务建好。

## 核心功能节点

### 1. 用户与登录

- 用户注册、登录、退出。
- 密码登录，后续可扩展 OAuth。
- 登录后返回访问令牌和当前用户信息。
- 用户可以属于多个工作区。
- 默认工作区可以是个人，也可以是公司。

### 2. 工作区/租户

- 工作区代表一个租户，可以是个人或公司。
- 工作区下隔离知识库、内容、媒体账号、计划任务和审计日志。
- 公司工作区支持成员和角色。
- 角色建议：Owner、Admin、Editor、Viewer。
- 工作区可维护品牌名、行业、语言、默认语气、默认发布频率等配置。

### 3. 知识库

- 每个工作区维护独立知识库。
- 知识条目类型包括品牌资料、产品资料、案例、FAQ、风格指南、禁用词、目标受众。
- 生成文章时必须选择或自动检索知识上下文。
- 早期保存文本内容即可，后续增加 embedding 和 `pgvector` 检索。

### 4. 媒体平台维护

- 全局维护媒体平台类型，例如 WordPress、微信公众号、知乎、LinkedIn、X。
- 平台定义包含能力：是否支持文章、图片、定时发布、草稿箱、标签。
- 平台定义包含凭证字段 schema。
- 平台可启用/停用。
- 该能力属于平台管理后台，不属于普通租户工作台。

### 5. 平台账号资源维护

- 工作区绑定具体媒体账号。
- 保存账号名称、平台、外部账号标识、授权状态、凭证过期时间。
- 支持测试连接。
- MVP 阶段连接测试可以先 mock。
- 如果未来需要平台自有账号池，应独立设计，不混入租户媒体账号。

### 5.1 平台管理后台

- 管理后台是正式系统的必要组成，不应只做 demo 页面。
- 管理后台采用开源 `react-admin` 接入，和当前 React + MUI 技术栈一致。
- 平台管理员可查看全局用户、工作区、工作区成员关系。
- 平台管理员维护全局媒体渠道定义。
- 平台管理员查看各租户媒体账号资源和连接状态。
- 平台管理员后续应维护渠道适配器配置、渠道启停、凭证字段 schema 和审计日志。
- 普通租户用户只能维护自己工作区下的媒体账号，不能修改全局媒体渠道。

### 6. 内容生成工作流

- 输入关键词。
- 选择内容类型。
- 选择知识库。
- 生成请求记录。
- AI Provider 先用 mock，后续替换为真实模型。
- 生成后形成内容草稿。
- 内容状态包括 draft、review、approved、scheduled、published、failed、archived。

### 6.1 AI 接入方案

- 当前阶段采用 RAG + 写作 Skill 模板 + 结构化输出，不优先做微调。
- 租户知识库内容不用于微调模型，应按工作区、知识库、关键词和内容类型做检索召回。
- 不把整个知识库直接丢给模型，只选择相关知识片段 TopK 作为上下文。
- 知识条目需要做 chunk、embedding，并用 PostgreSQL + `pgvector` 做语义检索。
- 写文章能力抽象为 `WritingSkill`，包含写作目标、文章结构、语气、禁用规则、平台限制、输出 schema 和版本号。
- 后端抽象 `AIProvider`，早期实现 `mock` 和一个真实模型 Provider，避免一开始引入重 AI 框架。
- 生成流程为：关键词 -> 检索知识片段 -> 选择 WritingSkill -> 组装 prompt -> 调用 AIProvider -> 校验结构化输出 -> 保存草稿。
- AI 输出必须先进入 `draft`，不能直接发布。
- 每次生成必须记录 `GenerationRequest`，包含关键词、知识片段 ID、Skill 版本、Provider、模型、token/cost、原始输出、结构化结果和错误信息。
- 结构化输出建议返回标题、摘要、正文、关键词、章节、引用的知识条目 ID、风险提示等字段。
- 微调放到后期，在积累足够人工审核通过的高质量样本后，用于稳定格式、语气和特定文体，不用于存放租户私有知识。

### 7. 内容管理

- 内容列表。
- 新建内容。
- AI 生成内容。
- 编辑内容。
- 版本记录。
- 审核状态。
- 标签/分类。
- 平台适配字段。
- 内容预览和归档。

### 8. 发布计划

- 一次性计划：指定时间发布一篇。
- 周期性计划：每天、每周、每月定期发布。
- 计划规则包含频率、时间、目标账号、内容来源。
- 内容来源可以是手动选择内容、已审核内容池、自动生成内容。
- 计划支持启用和暂停。
- 计划维护下次执行时间。

### 9. 发布任务

- 发布计划生成具体任务。
- 任务状态包括 queued、running、succeeded、failed、retrying。
- 支持失败重试。
- 记录错误日志。
- 使用幂等键避免重复发布。
- 发布成功后回写外部 URL。

### 10. 审计与日志

- 登录日志。
- 成员操作日志。
- 内容修改日志。
- 发布任务日志。
- 凭证操作日志。
- AI 生成请求日志。

## 实施计划

### 阶段 1：产品骨架重构

- 将匿名租户 header 调整为登录用户 + 工作区上下文。
- 增加登录页和当前用户信息。
- 前端支持工作区切换。
- 后端所有业务接口从当前用户的工作区权限中获取租户上下文。
- 继续使用内存数据，便于快速验证产品流。

### 阶段 2：核心数据模型与数据库

- 接入 PostgreSQL。
- 增加数据库迁移。
- 建表：users、workspaces、workspace_members。
- 建表：knowledge_bases、knowledge_items。
- 建表：media_platforms、media_accounts。
- 建表：contents、content_versions。
- 建表：publish_schedules、publish_jobs、publish_results。
- 建表：audit_logs。

当前状态：

- 已提供 PostgreSQL + pgvector 的初始迁移。
- 已提供 Docker Compose 本地数据库。
- 后端需要可用的 `DATABASE_URL`，核心业务读写已从 PostgreSQL 加载并持久化。
- 当前数据库访问仍在 handler 附近，下一步需要拆出 repository/service 边界。

### 阶段 3：知识库和媒体账号

- 知识库 CRUD。
- 知识条目 CRUD。
- 媒体平台初始化数据仅保留小红书。
- 媒体账号绑定 CRUD。
- 连接测试接口先 mock。

当前状态：

- 已接入开源 `react-admin` 作为平台管理后台。
- Demo 用户 `demo@geopress.local` 是平台管理员。
- 平台后台支持查看用户、工作区、成员、小红书媒体平台和租户媒体账号资源。
- 平台后台媒体平台配置限制为小红书。
- 租户工作台继续负责知识库、媒体账号绑定、内容生成和发布计划等业务操作。

### 阶段 4：内容生成框架

- 关键词生成表单。
- 选择知识库。
- 生成请求记录。
- mock AI Provider。
- 生成草稿。
- 内容编辑和状态流转。
- 设计 `AIProvider` 接口，先保留 mock，再接入真实模型 API。
- 设计 `WritingSkill` 模板和版本字段，先内置常用文章模板。
- 设计结构化输出 schema，生成结果必须经过 JSON 校验后入库。
- 生成时保存 prompt 版本、上下文片段、模型信息和原始返回。

### 阶段 4.1：RAG 和真实 AI 接入

- 对知识条目做文本切分，生成知识片段。
- 为知识片段生成 embedding，写入 PostgreSQL + `pgvector`。
- 按工作区、知识库、关键词和内容类型检索 TopK 片段。
- 组装上下文时限制 token，保留知识片段 ID 以便追踪。
- 接入第一个真实 AI Provider。
- 将 AI 生成结果保存为草稿，进入人工编辑和审核流程。
- 记录 token、成本、延迟、失败原因和重试状态。

### 阶段 5：计划任务和发布任务

- 一次性发布计划。
- 周期性发布计划。
- schedule -> job 生成逻辑。
- 后台 worker。
- 发布适配器接口。
- mock 发布适配器。
- 任务日志和重试。

### 阶段 6：真实 AI 和真实平台

- 扩展更多 AI Provider。
- 根据审核通过样本评估是否需要微调。
- 增加相似内容查重、品牌一致性检查和禁用词检查。
- 接入第一个真实媒体平台，例如 WordPress。
- 再扩展微信公众号、知乎、LinkedIn 等。

## MVP 闭环

第一版产品闭环目标：

```text
登录 -> 进入工作区 -> 维护知识库 -> 维护媒体账号 -> 输入关键词生成 mock 草稿 -> 编辑内容 -> 创建发布计划 -> 生成发布任务 -> mock 发布成功
```
