1. 建立知识资产、分块和处理任务的数据模型
   把用户上传或录入的文件从“知识条目”中独立出来，作为可追踪的内容资产保存。范围包含资产元数据、所属知识库包关系、处理任务状态、分块内容、检索文本和 embedding 状态字段，为后续异步处理和语义检索提供稳定数据边界。

2. 实现默认规则处理链路
   上传后必须先完成可控、低成本的文本提取和规则分块，让普通用户不依赖 AI 也能维护知识资产。范围限定在 Word、文本、PDF 和图片的收敛入口、可支持格式的文本提取、规则 chunk、searchText 生成，以及不支持提取时的失败状态落库。

3. 提供知识资产工作区 API
   让前端可以创建、查看和追踪知识资产，而不是继续只维护零散知识条目。范围包含资产创建、资产列表、资产详情、chunk 列表、处理任务列表、知识库归属校验和 VIP AI 增强权限校验。

4. 接入生成时的知识资产分块检索
   内容生成只检索已处理的知识资产 chunk，不再维护知识条目和知识资产两套用户知识路径。范围包含工作区隔离、知识库包过滤、资产状态过滤、searchText/标题/标签/摘要/正文综合匹配、旧知识条目迁移为资产 chunk，以及 retrieved knowledge ID 的稳定记录。

5. 将 VIP AI 增强改为异步处理任务
   AI 增强是可选增值能力，不应阻塞资产基础入库和默认检索能力。范围包含增强任务入队、处理进度可见、成功后替换增强 chunk、失败后保留默认 chunk，以及队列不可用时的明确失败状态。

6. 更新租户工作台前端资产管理体验
   用户需要在知识库入口维护自己的内容资产，并看到上传、处理和增强进度。范围包含资产创建弹窗、文件/文本两种输入、知识库包多选、VIP 增强开关、资产列表、状态进度、详情中的 chunk 和任务展示。

7. 验证端到端合同和回归风险
   这次改动同时触及存储、处理、检索、生成和前端合同，需要用后端测试、前端构建/ lint 和 diff 检查确认主路径可用。范围包含新增 API 行为、VIP 权限、检索优先级、前端类型兼容和格式化检查，不把验证从对应交付单元中剥离成孤立工作。
